1Executive Summary

Vision · Goals · Scope

LocalDictation is a Windows 11 desktop application that adds a system-wide voice layer to the operating system. A single global hotkey captures speech from any application, transcribes it locally with Whisper, optionally refines it with a local LLM, and inserts the result into the currently-focused editable control — with no audio, transcript, or telemetry ever leaving the device.

1.1 Project Vision

Voice is the fastest human input modality, yet native Windows dictation is mediocre, cloud-tethered, and locked to specific apps. Commercial alternatives (Dragon, Wispr Flow, cloud dictation) either require subscriptions, ship audio to servers, or both. LocalDictation delivers a ChatGPT-Voice-quality experience that works everywhere — Teams, Slack, Notion, VS Code, Word, browsers, terminals — while keeping every byte on-device.

1.2 Goals & Non-Goals

🎯 Goals
  • Global hotkey capture from any focused control.
  • Fully local Whisper transcription (CPU-first).
  • Optional local LLM post-processing (grammar, rewrite, translate, format).
  • Reliable text insertion across heterogeneous Win32/UIA/browser controls.
  • Graceful floating editor fallback when insertion is impossible.
  • Clean, DI-driven, replaceable subsystems (speech, AI, output, plugins).
  • Searchable encrypted history; MSIX + installer distribution.
  • < 2 s perceived latency for a short utterance on a GPU-less laptop.
🚫 Non-Goals (v1)
  • No mandatory cloud services or accounts.
  • No real-time streaming captions/subtitles (batch dictation first).
  • No macOS/Linux port (architecture stays portable, but not shipped).
  • No speaker diarization or multi-speaker meetings.
  • No voice-command automation / OS control ("open Chrome").
  • No always-on wake-word listening (hotkey-gated by design, for privacy).
  • No mandatory telemetry.
Headline recommendation Ship on WPF (.NET 8) for UI, Whisper.NET (whisper.cpp backend) for speech with base/small as adaptive defaults, and Ollama as the pluggable local-LLM provider running Phi-3.5-mini / Qwen2.5-3B quantized. Every one of these is behind an interface and swappable. Details and justifications in §6 Technology Decisions.

/// <summary>Transcribes captured audio into text, locally.</summary>
/// <remarks>Implementations: Whisper.NET (default), FasterWhisper (opt), Cloud (plugin).</remarks>
public interface ISpeechEngine : IAsyncDisposable
{
    string Name { get; }
    bool IsReady { get; }
    Task WarmUpAsync(CancellationToken ct);
    Task<Result<Transcript>> TranscribeAsync(AudioClip clip, TranscribeOptions opt, CancellationToken ct);
}

/// <summary>Optionally refines transcribed text (grammar, rewrite, translate…).</summary>
/// <remarks>NoOpTextProcessor lets the app work with zero LLM installed (FR-10).</remarks>
public interface ITextProcessor
{
    bool IsAvailable { get; }
    Task<Result<string>> ProcessAsync(string text, ProcessingMode mode, CancellationToken ct);
}

/// <summary>Delivers final text to a destination (focused control, editor, clipboard…).</summary>
public interface IOutputTarget
{
    int Priority { get; }
    bool CanHandle(TargetControl target);
    Task<OutputResult> DeliverAsync(string text, TargetControl target, CancellationToken ct);
}
Why Result<T> not exceptions on the hot path The dictation pipeline expects predictable failure (no mic, LLM offline, insertion blocked). Modeling these as Result<T> keeps control flow explicit and testable; only truly exceptional faults throw. This is a consistent, single error-handling pattern (per code-quality guidance).

6.3.3 Latency Optimization Strategies

🔥 Model warm-up

Fire a 1-token dummy generation on idle after first activation so weights + KV cache are hot before the user's first real request.

🧲 Keep resident

Set Ollama keep_alive so the model stays in RAM between dictations; unload on configurable idle timeout to reclaim memory.

💤 Lazy loading

Never load the LLM at app start. Load on first AI-enabled dictation (or background-preload if user opts in).

🤖 Auto model select

Probe CPU/RAM/power; pick tier at startup and re-evaluate when unplugged (battery-aware downshift).

🚀 Background init

Whisper model + Ollama connectivity probed on a background thread during tray startup; UI never blocks.

♻️ Reuse & cache

Single long-lived engine instance; prompt templates precompiled; identical (text,mode) results memoized briefly for retry.

7Windows Integration Deep-Dive

Hotkeys · Focus · Insertion

7.1 Global Hotkeys — RegisterHotKey vs Low-Level Hook

RegisterHotKey (user32)
Pros
  • OS-managed, near-zero CPU overhead.
  • No global keyboard interception → no AV/EDR flags.
  • Delivers WM_HOTKEY reliably even when app is background.
Cons
  • Limited combos; can collide with other apps' registrations.
  • Can't do push-to-talk hold semantics cleanly.
  • No access to raw key stream.
Low-level hook (WH_KEYBOARD_LL)
Pros
  • Any combo, push-to-talk hold, chords, double-tap.
  • Full control over suppression.
Cons
  • Runs on every keystroke → perf & privacy scrutiny; must be lean.
  • Frequently flagged by AV/EDR as keylogger-like.
  • Callback must return fast or Windows silently removes the hook.
Recommendation: RegisterHotKey primary, opt-in hook for push-to-talk Default to RegisterHotKey (e.g. Ctrl+Win+Space) for privacy, safety, and reliability. Offer an opt-in low-level-hook mode only for users who want push-to-talk hold, with a clear explanation. Both implement IHotkeyService.

7.4 Text Insertion Strategy Chain

A prioritized chain of responsibility: try the most reliable applicable strategy, fall back down the chain, and finally to the floating editor.

StrategyMechanismReliabilityTrade-offs
1. UIA ValuePattern.SetValueSets control value directlyHigh where supportedOverwrites (not caret-insert); many controls don't implement it.
2. Clipboard pasteSave clip → set text → SendInput Ctrl+V → restoreHighest general-purposeClobbers clipboard momentarily; must save/restore; some apps block programmatic paste.
3. SendInput unicodeSynthesize KEYEVENTF_UNICODE keystrokesGoodSlow for long text; affected by IME/keyboard state; respects caret.
4. UIA TextPattern insertAccessibility text insertionVariesSparse app support.
5. Floating editorPresent text for manual copyAlways worksNot automatic; user does one paste.
Default order Clipboard-paste first for rich/browser/terminal targets (most reliable in practice), SendInput for simple edit fields where clipboard clobber is undesirable, then floating editor. Order is policy-configurable per app via plugins. Always save & restore the prior clipboard and honor a "never touch clipboard" setting.

7.5 Overlay, Tray, Startup, DPI & Multi-Monitor

🪟 Overlay window

Borderless, Topmost, WS_EX_NOACTIVATE + WS_EX_TOOLWINDOW so it never steals focus or appears in Alt-Tab. Positioned near the target window on the correct monitor.

📍 Multi-monitor / DPI

Per-Monitor-V2 DPI awareness in the app manifest. Overlay placement uses MonitorFromWindow of the target and that monitor's DPI. Survives virtual-desktop switches (overlay follows active desktop).

🔔 Tray + notifications

H.NotifyIcon for tray menu; toast notifications via Windows App SDK / ToastNotificationManager for errors and status.

🚀 Startup with Windows

MSIX: StartupTask manifest entry (user-toggleable in Windows Settings). Unpackaged: Run registry key or Startup folder shortcut.

🛡️ UAC / admin

Run as standard user by default. Note: standard-user apps cannot SendInput into elevated windows (UIPI). Detect elevated target → advise, offer floating editor. Optional elevated mode documented but not default (least privilege).

🖱️ Click-through

Overlay ignores mouse via WS_EX_TRANSPARENT except for its ESC/stop hotspot, so it never interferes with the underlying app.

8Audio Pipeline

Capture · VAD · Buffering
◈ Mermaid — audio pipelinemermaid
flowchart LR
  MIC[Mic · WASAPI via NAudio] --> RES[Resample → 16kHz mono float32]
  RES --> NS[Optional noise suppression · RNNoise]
  NS --> VAD[VAD · Silero/WebRTC]
  VAD -->|speech| RING[(Ring Buffer)]
  VAD -->|silence > N ms| STOP[Auto-stop]
  RING --> CLIP[AudioClip]
  STOP --> CLIP
  CLIP --> ASR[Whisper]
  CANCEL((ESC)) -.-> DISCARD[Discard buffer]
🎚️ Capture

NAudio WASAPI shared-mode capture. Enumerate devices via MMDeviceEnumerator; honor user-selected mic; hot-swap on default-device change events.

🔁 Format

Whisper wants 16 kHz, mono, 32-bit float. Resample in-pipeline (WdlResampler). Keep raw only in memory; no disk unless the user enables audio-retention.

🗣️ VAD

Silero VAD (ONNX) preferred (robust), WebRTC VAD as lightweight fallback. Drives auto-stop after configurable trailing silence (default 800 ms) and trims leading silence.

🧹 Noise suppression

Optional RNNoise pass for noisy environments; off by default (adds latency). Behind IAudioFilter.

📦 Buffering

Lock-free ring buffer decouples capture thread from consumers. Batch (record-then-transcribe) for v1; architecture leaves room for chunked-streaming later.

🛑 Cancel & recovery

ESC/timeout → cancel token discards buffer, releases device. Device errors (unplugged mic) surface as Result failure with a clear toast; pipeline tears down cleanly.

Latency Streaming capture means the audio is already fully buffered the instant the user stops → Whisper starts immediately. Combined with a warm model, a 4–6 s utterance transcribes in well under a second on base.

10Configuration & History Systems

Settings · SQLite

10.1 Configuration System

🗄️ Storage

JSON at %AppData%/LocalDictation/settings.json via ISettingsStore. Secrets (e.g. optional cloud plugin keys) encrypted with DPAPI (CurrentUser). Atomic writes (temp+rename).

🔀 Migration

Versioned schema with SchemaVersion + ordered migrators; unknown-future settings preserved (round-trip safe).

Covers: hotkeys, microphone, Whisper model, LLM provider/model, language, theme, overlay, history retention, startup, GPU preference, prompt templates, text-processing rules, and per-plugin settings — all strongly typed as AppSettings with change notifications so services react live.

10.2 History System (SQLite)

Recommendation: SQLite via Microsoft.Data.Sqlite + Dapper/EF Core Embedded, zero-config, transactional, FTS5 for search. File at %AppData%/LocalDictation/history.db.

🧵 Threading model

UI dispatcher for VM/overlay only. Audio capture on dedicated thread. Whisper/LLM inference on background Tasks. All I/O async; CancellationToken threaded end-to-end (ESC cancels everything).

❄️ Cold vs warm start

Cold: tray icon + DI ready fast; models NOT loaded. Warm: first activation triggers background model load with progress; subsequent activations are instant.

♻️ Model reuse

Single resident Whisper context; Ollama keep_alive. Idle-timeout unload reclaims RAM. Avoid per-dictation reloads (the #1 latency killer).

📉 CPU utilization

Cap Whisper threads to physical cores −1 to keep the machine responsive during dictation; battery-aware downshift on unplug.

13Security & Privacy

Threat model · Least privilege
🔌 Offline by default

Core app opens no outbound sockets. Verified by an integration test asserting zero network handles during a dictation. Model downloads are explicit, user-initiated, over HTTPS with checksum verification.

🔐 Encryption at rest

Settings secrets via DPAPI; history optionally SQLCipher/DPAPI-encrypted. No plaintext secrets on disk.

🙈 Sensitive input detection

UIA IsPassword → hard block. Configurable app blocklist (banking, password managers). "Never dictate here" per-app rule.

📋 Clipboard hygiene

Save/restore prior clipboard around paste; optional "never touch clipboard" mode; clear transcript from clipboard after configurable delay.

🧩 Plugin security

Signature checks, capability declarations, scoped context, no ambient network. Untrusted → future out-of-process sandbox.

🛡️ Least privilege

Runs as standard user; no admin required; minimal MSIX capabilities (mic + runFullTrust only).

🧪 Unit

xUnit + Moq. Domain logic, pipeline orchestration, strategy selection, settings migration. Mock all ports. Target ≥70% on Application/Domain.

🔗 Integration

Real Whisper on fixed WAV fixtures → assert transcript ≈ expected (WER threshold). Real SQLite history. Ollama gated behind availability check.

Golden-transcript corpus; snapshot tests on prompt-template outputs (deterministic temp=0).

18Design Decisions (ADRs)

Problem · Options · Choice
ADR-01UI Framework = WPF
Problem

Need a Windows UI with deep Win32 interop for global overlays, hooks, UIA.

Options

WPF · WinUI 3 · Avalonia.

Decision & reasoning

WPF — best-in-class, best-documented Win32 interop and topmost/no-activate overlay control; largest ecosystem. See §6.1.

Trade-offs

Windows-only; mitigated by thin MVVM presentation to keep Avalonia as a future port path.

Future alternative

Avalonia if cross-platform becomes a goal.

ADR-02Speech = Whisper.NET (whisper.cpp)
Problem

Local, offline, CPU-fast, easy-to-deploy speech recognition in .NET.

Options

Whisper.NET · whisper.cpp direct · Faster-Whisper · ONNX · PyTorch.

Decision

Whisper.NET — managed NuGet over fastest CPU backend, GGUF-as-file deployment, async API. See §6.2.

Trade-offs

Native binary shipping; no Python-ecosystem tooling. Acceptable for offline single-file deploy.

Future alternative

ONNX + DirectML engine for integrated-GPU acceleration (same port).

ADR-03Local LLM = Ollama default + LlamaSharp embedded
Problem

Optional local AI enhancement without cloud dependency, robust on CPU laptops.

Options

Ollama · llama.cpp/LlamaSharp · ONNX GenAI · cloud.

Decision

Ollama default (out-of-process crash isolation + model manager) with embedded LlamaSharp for no-second-install users; both behind ITextProcessor; NoOp when none. See §6.3.

Trade-offs

Ollama is a separate install; solved by embedded fallback.

Future alternative

ONNX GenAI + DirectML; cloud providers as opt-in plugins only.

ADR-04Hotkey = RegisterHotKey (hook opt-in)
Problem

Reliable global activation without keylogger-like behavior.

Decision

RegisterHotKey by default (privacy, AV-safe, low overhead); low-level hook opt-in only for push-to-talk. See §7.1.

Trade-offs

Fewer combos / no hold semantics by default — acceptable for privacy-first posture.

ADR-05Insertion = clipboard-first strategy chain
Problem

Insert reliably across wildly different controls.

Decision

Prioritized chain (UIA/clipboard/SendInput) with clipboard save-restore and floating-editor fallback; per-app order via plugins. See §7.4.

Trade-offs

Momentary clipboard use; mitigated by save/restore + opt-out.

ADR-06History store = SQLite + FTS5
Problem

Searchable, durable, encryptable local history.

Decision

SQLite (Microsoft.Data.Sqlite) with FTS5 search, WAL, optional SQLCipher/DPAPI encryption. See §10.2.

Trade-offs

Encryption UX complexity; opt-in.
